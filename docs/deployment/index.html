Schema d'architecture

Internet | [ DNS: domaine.com ] ──> [ IP du VPS ] | ┌──────┴──────┐ │ Port 80 │ ← Redirection HTTP → HTTPS │ Port 443 │ ← TLS (Let's Encrypt auto) └──────┬──────┘ | ┌─────────┴─────────────────────────────┐ │ CADDY (FrankenPHP) │ │ Reverse proxy + PHP + TLS auto │ │ │ │ /oidc/* ──> Keycloak:8080 │ │ @pwa ──> PWA:3000 (Next.js) │ │ php_server ──> Symfony API │ │ /images/* ──> fichiers statiques │ │ /media/* ──> fichiers uploads │ └───┬──────────────┬────────────────┬────┘ │ │ │ ┌──────┴──┐ ┌──────┴──┐ ┌────────┴────────┐ │ Keycloak │ │ PWA │ │ PostgreSQL │ │ :8080 │ │ :3000 │ │ (managed DB │ │ │ │ Next.js │ │ ou local) │ └────┬─────┘ └─────────┘ └─────────────────┘ │ ┌──────┴──────┐ │ KC Postgres │ │ (interne) │ └─────────────┘

Flux des requetes

  1. Navigateur → https://domaine.com → Caddy (port 443)
  2. Caddy verifie le type de requete :
    • HTML demande → PWA (Next.js SSR)
    • API (/docs, /graphql, JSON) → Symfony (php_server)
    • /oidc/* → Keycloak (authentification)
  3. Symfony communique avec PostgreSQL via DATABASE_URL
  4. Keycloak gere l'authentification OIDC avec son propre PostgreSQL interne

Pre-requis : DNS & Domaine

Critique — A faire AVANT le deploiement Le certificat SSL Let's Encrypt ne sera emis que si le DNS pointe deja vers l'IP du serveur. Configurer le DNS en premier.

Verification DNS

# Depuis votre machine locale nslookup staging.creazot.com # Doit retourner l'IP du VPS # Depuis le VPS nslookup staging.creazot.com # Meme IP

SSL / HTTPS

Caddy obtient automatiquement un certificat Let's Encrypt si :

  • ☑ Le DNS pointe vers l'IP du serveur
  • ☑ Les ports 80 et 443 sont ouverts et mappes directement (pas 8081→80)
  • ☑ SERVER_NAME contient le domaine sans prefixe http://
Piege frequent SERVER_NAME=http://staging.creazot.com → force HTTP seulement, pas de certificat SSL.
SERVER_NAME=staging.creazot.com → Caddy active HTTPS automatiquement.

Pre-requis : Base de donnees

Option A : PostgreSQL local (Docker)

Le compose.yaml inclut un service database PostgreSQL. Rien a configurer en externe.

Option B : PostgreSQL manage (DigitalOcean)

Si vous utilisez une base de donnees geree par DigitalOcean :

3 verifications obligatoires
1

Trusted Sources

Ajouter l'IP du Droplet dans Databases → Settings → Trusted Sources. Sans cela, la connexion sera refusee.

2

Hostname correct

Copier le hostname exact depuis Databases → Connection Details → host. Il ressemble a :

db-postgresql-xxx-do-user-xxxxx-0.x.db.ondigitalocean.com

Verifier que ce hostname est resolvable depuis le VPS :

nc -zv -w 5 <HOSTNAME> 25060
3

Mot de passe reel

Dans le dashboard DigitalOcean, cliquer sur Show pour reveler le vrai mot de passe. Ne pas copier le texte show-password qui est un placeholder.

DATABASE_URL pour une BDD managee

DATABASE_URL=postgresql://USER:PASSWORD@HOSTNAME:PORT/DBNAME?serverVersion=16&charset=utf8&sslmode=require
sslmode=require Obligatoire pour les bases de donnees managees DigitalOcean. Sans cela, la connexion est refusee.

Deploiement rapide

Objectif : deployer en moins de 2 minutes Ce guide suppose que les pre-requis (VPS, DNS, base de donnees) sont deja en place.

Commandes de deploiement

# 1. Se connecter au VPS ssh root@<IP_DU_VPS> # 2. Cloner le depot (premiere fois uniquement) git clone https://github.com/Seb974/achats-harel.git ~/achats-harel cd ~/achats-harel git checkout 4.0 # 3. Copier et configurer le .env (voir section suivante) cp .env.example .env nano .env # 4. Supprimer le compose.override.yaml (DEV) pour la prod rm -f compose.override.yaml # 5. cd ~/achats-harel git pull origin 4.0 docker compose -f compose.yaml -f compose.prod.yaml build --no-cache docker compose -f compose.yaml -f compose.prod.yaml up -d --force-recreate

Build & Lancement

Commandes de build

# Build complet (premiere installation ou changement Dockerfile) docker compose -f compose.yaml -f compose.prod.yaml build --no-cache # Build avec cache (mises a jour de code) docker compose -f compose.yaml -f compose.prod.yaml build # Lancement docker compose -f compose.yaml -f compose.prod.yaml up -d
Temps de build Un build --no-cache prend 30-40 minutes sur un VPS 2 vCPU (compilation des extensions PHP : intl, imagick, etc.). Avec cache, c'est beaucoup plus rapide.

Checklist post-deploiement

Verification des services

# 1. Tous les conteneurs sont "Up" et "healthy" docker compose ps # 2. PHP peut joindre la BDD docker compose exec php php bin/console dbal:run-sql "SELECT 1" # 3. Caddy sert le bon certificat SSL echo | openssl s_client -connect localhost:443 -servername staging.creazot.com 2>/dev/null | \ openssl x509 -noout -issuer -subject -dates # issuer doit contenir "Let's Encrypt" # 4. Keycloak repond en interne curl -sI http://localhost:8080/admin/master/console/ # Doit retourner HTTP 200 # 5. Keycloak repond via Caddy curl -sk https://staging.creazot.com/oidc/admin/master/console/ | head -3 # Doit retourner du HTML # 6. La PWA repond curl -sk -o /dev/null -w '%{http_code}' https://staging.creazot.com/ # Doit retourner 200 # 7. L'API repond curl -sk https://staging.creazot.com/docs.jsonld # Doit retourner du JSON-LD

Checklist complete

  • ☐ DNS pointe vers l'IP du VPS
  • ☐ Ports 80 et 443 mappes directement (pas 8081/8443)
  • ☐ SERVER_NAME sans prefixe http://
  • ☐ compose.override.yaml supprime ou renomme
  • ☐ DATABASE_URL avec le bon hostname, bon mot de passe, sslmode=require
  • ☐ POSTGRES_DB = nom de la BDD (pas le hostname !)
  • ☐ Trusted Sources configure sur DigitalOcean (si BDD managee)
  • ☐ Tous les conteneurs Up et healthy
  • ☐ Certificat SSL = Let's Encrypt (verifier depuis le serveur)
  • ☐ Realm Keycloak cree (creazot)
  • ☐ Console admin Keycloak accessible via /oidc/admin/

Diagnostic des erreurs

Guide base sur les erreurs reellement rencontrees lors du deploiement de mars 2026.

PHP   Conteneur en boucle "Tentative X/30..."

Symptome

Le conteneur PHP redemarre en boucle, les logs montrent des tentatives de connexion a PostgreSQL.

Causes possibles (dans l'ordre de frequence)

  1. Hostname de la BDD incorrect dans DATABASE_URL — verifier avec nc -zv -w 5 HOSTNAME PORT
  2. Trusted Sources non configure sur DigitalOcean — ajouter l'IP du VPS
  3. Mot de passe incorrect — copier le vrai mot de passe (cliquer "Show")
  4. sslmode=require manquant — obligatoire pour les BDD managees

Commandes de diagnostic

# Tester la resolution DNS du hostname nc -zv -w 5 <DB_HOSTNAME> <PORT> # Verifier les variables dans le conteneur docker compose exec php env | grep DATABASE # Verifier les logs docker compose logs php --tail 30
PHP   "password authentication failed for user"

Symptome

PostgreSQL est joignable mais refuse l'authentification.

Solution

  1. Verifier que POSTGRES_PASSWORD et le mot de passe dans DATABASE_URL sont identiques
  2. Verifier que c'est le vrai mot de passe (pas "show-password")
  3. Apres modification du .env, recreer le conteneur avec --force-recreate
ENV   Modification du .env non prise en compte

Symptome

Vous modifiez le .env mais le conteneur utilise toujours les anciennes valeurs.

Cause

docker compose restart ne recharge pas le .env. Il faut recreer le conteneur.

Solution

# Recreer le conteneur pour charger le nouveau .env docker compose -f compose.yaml -f compose.prod.yaml up -d --force-recreate php # Verifier que les nouvelles valeurs sont bien chargees docker compose exec php env | grep DATABASE
SSL   HTTPS barre en rouge / certificat invalide

Diagnostic

# Verifier le certificat directement sur le serveur echo | openssl s_client -connect localhost:443 -servername DOMAINE 2>/dev/null | \ openssl x509 -noout -issuer # Si issuer = "Let's Encrypt" → le serveur est OK, probleme cote client # Si issuer = "Caddy Local Authority" → SSL auto pas active

Si le certificat est auto-signe (Caddy Local Authority)

  • Verifier que SERVER_NAME ne commence pas par http://
  • Verifier que les ports 80 et 443 sont mappes directement
  • Verifier que le DNS pointe vers le serveur

Si le certificat est Let's Encrypt mais le navigateur montre une erreur

Un pare-feu d'entreprise (Fortinet, Palo Alto, etc.) intercepte le trafic SSL et remplace le certificat. Tester depuis un autre reseau (4G, VPN).

502   Bad Gateway sur Keycloak (/oidc/*)

Diagnostic

# Keycloak tourne-t-il ? docker compose ps keycloak # Keycloak repond-il en interne ? curl -sI http://localhost:8080/admin/master/console/ # Keycloak repond-il via Caddy ? curl -sk https://DOMAINE/oidc/admin/master/console/ | head -3 # Verifier les erreurs Keycloak docker compose logs keycloak --tail 20 | grep -i error

Causes frequentes

  • Keycloak pas encore demarre — prend ~15s, attendre
  • KC_PROXY: edge (deprecie) — remplacer par KC_PROXY_HEADERS: xforwarded
  • Erreur "Failed to parse port" — probleme de forwarded headers
  • Firewall SSL (Fortinet) — le serveur repond OK mais le client recoit un 502 a cause de l'interception SSL
DNS   "Name or service not known"

Symptome

Le hostname de la base de donnees ne se resout pas depuis le VPS.

Solution

  • Verifier que le hostname est exactement celui indique dans le dashboard DigitalOcean
  • Tester la resolution : nc -zv -w 5 HOSTNAME PORT
  • Si la BDD vient d'etre creee, attendre quelques minutes pour la propagation DNS
PERF   Build Docker extremement lent

Explication

Le build --no-cache compile les extensions PHP (intl, imagick, pdo_pgsql) depuis les sources. Cela prend 30-40 minutes sur un petit VPS.

Optimisations

  • Ne pas utiliser --no-cache sauf si necessaire
  • Utiliser un VPS avec plus de CPU pour le build
  • Envisager de pre-builder les images et les pousser sur un registry
SSH   Timeout de connexion SSH

Cause

Le VPS est surcharge (CPU 100%, build en cours, conteneurs qui redemarrent en boucle).

Solutions

  • Attendre 30-60 secondes et reessayer
  • Verifier avec ping <IP> que le serveur repond
  • Acceder via la console web DigitalOcean si SSH est impossible

Retour d'experience — Mars 2026

Chronologie des problemes rencontres lors du premier deploiement sur DigitalOcean et lecons apprises.

Erreur 1 Mauvais hostname de base de donnees

Probleme : Le hostname PostgreSQL dans DATABASE_URL (staging-database-do-user-...) ne correspondait pas au vrai hostname de la BDD managee (ah-chou-database-do-user-...).

Symptome : nc: getaddrinfo: Name or service not known

Lecon : Toujours copier le hostname depuis le dashboard DigitalOcean → Databases → Connection Details et tester avec nc -zv.

Erreur 2 POSTGRES_DB confondu avec le hostname

Probleme : POSTGRES_DB a ete rempli avec le hostname de la BDD au lieu du nom de la base (defaultdb).

Lecon : POSTGRES_DB = nom de la base (ex: defaultdb, app). Ce n'est jamais une URL ou un hostname.

Erreur 3 Faux mot de passe ("show-password")

Probleme : Le mot de passe copie depuis le dashboard DigitalOcean etait le placeholder show-password au lieu du vrai mot de passe.

Symptome : FATAL: password authentication failed for user "doadmin"

Lecon : Toujours cliquer sur "Show" ou l'icone oeil pour reveler le vrai mot de passe avant de le copier.

Erreur 4 restart au lieu de force-recreate

Probleme : Apres modification du .env, un docker compose restart a ete utilise. Les anciennes variables etaient toujours actives dans le conteneur.

Lecon : restart ≠ up -d --force-recreate. Seul force-recreate recharge les variables d'environnement.

Erreur 5 SERVER_NAME avec prefixe http://

Probleme : SERVER_NAME=http://staging.creazot.com empechait Caddy d'activer HTTPS et d'obtenir un certificat Let's Encrypt.

Lecon : SERVER_NAME ne doit contenir que le domaine, sans protocole : staging.creazot.com.

Erreur 6 Ports 8081/8443 au lieu de 80/443

Probleme : Les ports etaient mappes 8081:80 et 8443:443, empechant Let's Encrypt de valider le domaine via le challenge HTTP sur le port 80.

Lecon : En production, toujours mapper 80:80 et 443:443.

Erreur 7 HTTPS barre malgre un certificat valide

Probleme : Le certificat Let's Encrypt etait valide sur le serveur, mais un pare-feu Fortinet sur le reseau local remplacait le certificat (SSL inspection).

Lecon : Toujours verifier le certificat depuis le serveur (openssl s_client) pour isoler un probleme serveur vs. reseau. Tester depuis un reseau different si necessaire.

Regles d'or du deploiement

  1. DNS d'abord — Toujours configurer le DNS avant de deployer
  2. Tester la BDD avant de lancer les conteneurs — nc -zv HOSTNAME PORT
  3. Verifier le .env 3 fois — La majorite des erreurs viennent de la
  4. force-recreate apres tout changement de .env — restart ne suffit pas
  5. Diagnostiquer depuis le serveur — openssl, curl, docker exec env
  6. Pas de compose.override.yaml en prod — Le supprimer ou le renommer
