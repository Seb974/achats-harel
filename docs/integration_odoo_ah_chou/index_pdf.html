Gestion des Listes de Prix (Tarifs)

Structure tarifaire

6 niveaux de tarifs (T1 à T6) plus des tarifs spécifiques.

Hiérarchie d'attribution

CATÉGORIE (GMS, CHR...) → Tarif par défaut ↓ peut être surchargé par GROUPE CLIENT (GBH, SMDIS...) → Tarif groupe ↓ peut être surchargé par ENSEIGNE (Carrefour, U...) → Tarif enseigne ↓ peut être surchargé par TYPOLOGIE (Hyper, Super...) → Tarif typologie ↓ peut être surchargé par CLIENT → Tarif spécifique

Intégration Module Achats Harel 4.0

Le module Achats Harel 4.0 est une application web PWA qui fonctionne comme une "calculette" pour les achats internationaux.

Architecture

┌─────────────────────────────────────────┐ │ ODOO 19 ENTERPRISE │ │ ┌─────────────────────────────────┐ │ │ │ Produits / Fournisseurs │◄──┼─── Données maîtres │ │ Commandes d'achat │◄──┼─── Création commandes │ └─────────────────────────────────┘ │ └────────────────▲────────────────────────┘ │ API XML-RPC ┌────────────────┴────────────────────────┐ │ MODULE ACHATS HAREL 4.0 │ │ ┌─────────────────────────────────┐ │ │ │ Récupère: produits, UoM, frs │ │ │ │ Calcule: PR en EUR │ │ │ │ Crée: commandes confirmées │ │ │ └─────────────────────────────────┘ │ └─────────────────────────────────────────┘

Flux de données

DirectionDonnéesFréquence

Odoo → AppFournisseurs (id, nom)À la demande
App → OdooCommande d'achat confirméeAprès validation

Calcul Prix de Revient

PR (EUR) = Prix USD × Parité EUR/USD × Coef total approche Exemple: Prix USD/UVC: 5.00 Parité: 0.92 Coef approche: 1.35 PR EUR = 5.00 × 0.92 × 1.35 = 6.21 EUR

Mode Dry-Run

Par défaut, DRY_RUN = True dans config.py. Cela permet de simuler les imports sans écrire réellement dans Odoo.

# Pour activer l'import réel: DRY_RUN = False

Prérequis Odoo

Important: Avant de lancer les imports, les éléments suivants doivent être créés dans Odoo:
  • Champs personnalisés clients (x_groupe_client, x_enseigne, etc.)
  • Champs personnalisés produits (x_zone_peche, x_nom_scientifique, etc.)
  • Champs personnalisés fournisseurs (x_groupe_fournisseur, x_is_transitaire, etc.)
  • Listes de prix (Tarif Base, T1 à T6)
  • Taxes TVA DOM (2.1%, 8.5%, Exonéré)
  • Conditions de paiement

  2. Clients - Crée automatiquement les catégories

3.1 - Logique de tarification

Principe : Prix de vente = Prix d'achat (coût) × (1 + Marge%)

Flux de données :

  1. L'application Achats Harel crée une commande fournisseur dans Odoo avec le prix d'achat

  3. Les listes de prix calculent automatiquement le prix de vente en appliquant la marge
  4. Chaque client est associé à une liste de prix (via le champ property_product_pricelist)

5.5 - Guide FEFO : Comment ça marche concrètement ?

🎯 FEFO = First Expiry First Out
"Premier à expirer = Premier à sortir" - La stratégie optimale pour les produits périssables

📥 1. À la RÉCEPTION des marchandises

ÉtapeAction utilisateurCe que fait Odoo
1 Valider la réception Demande de créer/sélectionner un numéro de lot
2 Créer le lot (ex: LOT-AAAA-MM-JJ-001) Affiche les champs de dates d'expiration
3 Saisir la Date d'expiration (DLC/DLUO) Enregistre la date sur le lot
4 Ranger dans le congélateur Stock disponible avec date d'expiration connue
💡 Astuce : Utilisez un format de lot incluant la date de réception pour faciliter le suivi (ex: LOT-AAAA-MM-JJ-XXX)

📤 2. À la PRÉPARATION de commande (picking)

SituationCe que fait FEFO automatiquement
Client commande 10 cartons de crevettes Odoo propose automatiquement les lots dont la date d'expiration est la plus proche
Plusieurs lots disponibles Tri par date d'expiration croissante (le plus proche en premier)
Stock insuffisant sur un lot Complète avec le lot suivant (toujours par ordre d'expiration)
📋 Exemple concret : Stock CONGELATEUR-1 - Crevettes 1kg : ┌─────────────────────┬────────────┬──────────────┐ │ Lot │ Quantité │ Expiration │ ├─────────────────────┼────────────┼──────────────┤ │ LOT-001 │ 50 cartons │ Expire en 1er│ ← FEFO prend celui-ci en 1er │ LOT-002 │ 80 cartons │ Expire en 2ème│ ← puis celui-ci │ LOT-003 │ 100 cartons│ Expire en 3ème│ ← puis celui-ci └─────────────────────┴────────────┴──────────────┘ Commande client : 70 cartons FEFO sélectionne automatiquement : → 50 cartons du LOT-2025-11-15-001 (expire le plus tôt) → 20 cartons du LOT-2025-12-20-002 (complément)

⚠️ 3. ALERTES de péremption

ChampDescriptionUsage

5.6 - Flux de Stock : Parcours des marchandises

🚢 FLUX IMPORT - Commande fournisseur étranger

📝 Commande achat validée ↓ 🚢 TRANSIT MARITIME/EN MER ←────── Départ du port d'origine ↓ (notification arrivée) ⚓ TRANSIT MARITIME/PORT ARRIVEE ── Arrivée au port de La Réunion ↓ 📋 TRANSIT MARITIME/DEDOUANEMENT ── Documents douane ↓ (mainlevée obtenue) 📥 RECEPTION ─────────────────────── Contrôle visuel, température, état ↓ ├── ⚠️ QUARANTAINE (si problème détecté) │ ↓ (après validation) │ ↓ └── ❄️ CONGELATEUR-1/2/3 ou 🥶 REFRIGERE (rangement final)
💡 Avantage : À tout moment, vous savez exactement où est votre marchandise : en mer, au port, en dédouanement, ou en stock.

🏭 FLUX LOGISTISUD - Stock déporté

📥 RECEPTION (arrivée directe chez Logistisud) ↓ 🏭 LOGISTISUD/CONGELATEUR ou REFRIGERE ── Stockage externe ↓ (commande client nécessitant ce stock) 📦 Transfert vers AH CHOU ────────────── Rapatriement ↓ ❄️ CONGELATEUR ou 🛒 PICKING ──────────── Préparation
📊 Comptabilité stock : Le stock Logistisud est comptabilisé dans votre stock total mais vous pouvez filtrer pour voir uniquement le stock "sur site AH CHOU".

📦 FLUX LIVRAISON - Commande client

📋 Commande client confirmée ↓ ❄️ CONGELATEUR-1/2/3 ─────────────── Sélection FIFO/FEFO automatique ↓ (lots les plus anciens/proches expiration) 🛒 PICKING ───────────────────────── Préparation commande (picking list) ↓ 📤 EXPEDITION ────────────────────── Contrôle final, étiquetage ↓ 🚚 → Client ──────────────────────── Bon de livraison signé

⚠️ FLUX QUARANTAINE - Gestion des anomalies

📥 Produit reçu avec problème détecté (température, emballage, DLC courte) ↓ ⚠️ QUARANTAINE ───────────────────── Blocage temporaire ↓ ├── ✅ Produit OK après contrôle → Transfert vers CONGELATEUR ├── 🔄 Retour fournisseur ──────→ Transfert vers Vendors (retour) └── ❌ Destruction ─────────────→ Transfert vers Inventory Loss

6.2 - Problématique Métier : Codes-barres Dupliqués

⚠️ 78 produits non importés

Odoo n'accepte pas les codes-barres (EAN13) en doublon. Chaque code-barre doit être unique dans la base.

Causes identifiées dans les données

SituationExempleImpact
Même produit, origines différentes [204931] Tranches de Rouge Vielle Inde
[204948] Tranches de Rouge Vielle Vietnam
→ Même EAN: 3456470204931 		Le 2ème produit est rejeté
Produit vendu au kg (sans EAN propre) [203570] Vivaneau Madras Inde kg
[376] Vivaneau hublot Vietnam kg
→ EAN générique: 0200300000000 		Tous sauf le premier sont rejetés
Erreur de saisie historique Copier-coller d'un EAN existant lors de la création Collision imprévue

📋 Pourquoi c'est important pour le métier

À la caisse / en scan : Si deux produits ont le même code-barre, le scanner ne peut pas distinguer lequel vous vendez. Le mauvais produit (et donc le mauvais prix) peut être facturé.

En inventaire : Les mouvements de stock seront imputés au mauvais produit, faussant les quantités réelles.

En traçabilité : Impossible de retracer un lot spécifique si plusieurs produits partagent le même code.

6.5 - Procédure de Création des Variantes

Méthode manuelle dans Odoo

  1. Ouvrir le produit parent (celui qui a été importé avec succès)
  2. Aller dans l'onglet "Attributs & Variantes"
  3. Cliquer sur "Ajouter une ligne"
  4. Sélectionner l'attribut "Origine"
  5. Cocher les valeurs : Inde, Vietnam, Indonésie...
  6. Odoo crée automatiquement les variantes
  7. Sur chaque variante, renseigner le code-barre spécifique

Chemin dans Odoo

Ventes > Produits > [Sélectionner produit] > Onglet "Attributs & Variantes"

⚠️ Important : Pour les produits avec EAN générique (0200XXXXXXXXX), il est recommandé de :

  • Soit créer des codes-barres internes uniques
  • Soit laisser sans code-barre et utiliser la référence interne
